## 实施计划/方案文档维护规则

**目标**：确保每个功能模块的实施计划文档始终反映该功能的最新实现状态，不累积历史版本。

**规则说明**：

每次新功能开发完成或现有功能优化后，需要按以下步骤处理：

1. **分析改动涉及的模块**：明确本次改动影响的是哪个功能模块（如：视频解析、地图功能、用户收藏等）

2. **查找对应文档**：在 `需求文档&技术方案/` 目录下查找该模块是否有对应的实施计划或方案文档

3. **更新文档**：如果存在对应文档，将文档内容更新为本次实现的最新状态（覆盖式更新，不是追加）

4. **文档状态**：确保文档内容与当前代码实现一致，让用户能够通过文档快速了解该功能的最新情况

**文档更新原则**：
- 每次更新覆盖整个文档，而非在末尾添加变更记录
- 文档应当反映功能的当前状态（含所有已完成的部分）
- 如果功能仍在进行中，标注各部分的完成状态
- **必须同步记录后续待开发内容**：本次改动中未完成的部分（如二阶段功能、后续优化项），也需要写入对应文档的"待开发"或"未来计划"章节，确保文档完整反映该功能的全貌，而非只记录本次完成的部分
- **历史版本与本次改动整合**：更新文档时，不是简单用本次改动内容全部替换历史版本，而是结合历史版本的完整说明与本次新增/修改内容，合并为一份更齐全的完整说明。例如：历史版本已有详细的功能描述、流程说明，本次改动若只涉及局部修改，则保留其他未改动部分，只更新本次修改涉及的章节，形成一份连贯、完整的最新文档
- **验证清单同步更新**：本次新增或修改的功能，对应的验证清单也需要同步更新。验证清单要求结合历史版本的测试项与本次新增/修改内容，合并为一份齐全的完整验证清单，确保验证时不会遗漏之前已验证正常的部分

**示例**：
假设"收藏模块"之前只完成了一部分，现在新增了按标签分类的功能：
- 打开 `收藏模块功能调整与完善实施计划.md`
- 将文档中"按标签分类"部分从"待开发"更新为"已完成"
- 同时更新相关的技术实现说明、UI 描述等内容
- 删除所有旧版本的历史记录痕迹

**与产品功能清单的关系**：
- 实施计划文档：详细记录功能的技术实现、已完成状态、技术细节（供开发者参考）
- 产品功能清单：总览所有功能的完成状态，优先级，核心价值（供产品规划参考）
- 两者互补，功能完成后需同步更新

## 遇阻放弃规则

执行任务时，若某个问题尝试了 **3 种不同方法**仍无法解决，必须立即停止尝试，避免陷入死循环浪费 token。

**处理方式：**
1. 简要说明已尝试的 3 种方法及失败原因
2. 明确告知用户该问题当前无法自动解决
3. 给出建议的人工排查方向或下一步操作建议
4. 不再继续重试，等待用户指示

**适用场景举例：**
- 依赖安装失败（尝试不同版本、不同源、不同方式后仍报错）
- API 调用持续返回错误（尝试不同参数、不同端点后仍失败）
- 代码运行报错（尝试不同修复方案后仍无法通过）
- 文件读写权限问题（尝试不同路径、不同方式后仍失败）

## 产品功能清单维护规则

`需求文档&技术方案/产品功能清单.md` 是产品开发的核心牵引文档，记录所有已完成和待开发功能/优化项。

文档采用四张表格的结构：
- **已完成功能**：已上线的新功能开发
- **已完成优化**：已上线的功能增强和体验优化
- **未完成功能**：待开发的新功能
- **未完成优化**：待进行的优化项

**区分功能与优化项的定义：**
- **功能**：新增加的业务能力，通常涉及新页面、新交互、新数据模型
- **优化**：对现有功能的增强、改善体验、提升性能、准确率等，不增加新的业务边界

**更新时机：**
当涉及以下变更时，必须同步更新该文档：
- 新功能开发完成（从未完成功能移至已完成功能）
- 新优化项完成（从未完成优化移至已完成优化）
- 新增待开发功能（添加到未完成功能表格）
- 新增待优化项（添加到未完成优化表格）
- 功能或优化项调整/删除（更新对应表格行或删除）
- 技术实现方案变更（更新对应列）

**更新要求：**
- 保持四张表格的形式不变，分类清晰
- 每行包含：功能/优化名称、定位、核心解决的问题、关键要点、技术实现方案、优先级说明（未完成部分）
- 技术实现方案需简化说明，不超过一行
- 保持文档直观简洁，让人一眼看懂产品全貌
- 新增或调整功能/优化项时，必须评估其优先级，并按优先级从高到低排序插入对应表格
- 优先级评估维度：用户价值、业务紧迫性、实现难度（难度低加分）；若无明确指示，默认按此维度综合判断
- **特别注意**：明确区分功能与优化项——功能是新增能力，优化是增强现有能力

## 产品介绍维护规则

`需求文档&技术方案/产品介绍.md` 是面向用户的核心文档，让不了解产品的人能快速上手使用。当产品功能发生变化时，必须同步更新该文档。

**更新时机：**
当涉及以下变更时，必须检查并更新 `产品介绍.md`：
- 新功能上线 → 在「核心功能」新增功能说明
- 功能调整/优化 → 若影响用户体验方式，需同步更新
- 功能下线 → 从「核心功能」移除或注明已停用
- 使用流程变化 → 更新相关功能描述或 FAQ
- 常见问题解答 → 新增/修改 FAQ

**更新原则：**
- 保持结构稳定：仅在现有结构上补充或修正，不随意调整文档框架
- 语言简洁直观：面向普通用户，避免技术术语，用「你」称呼用户，每项功能一句话说清楚价值
- 与产品功能清单保持一致：新增功能应同步更新 `产品功能清单.md`

**文档结构（勿轻易调整）：**
- 产品定位 → 一句话说清产品是什么
- 核心功能 → 主要功能列表，每个功能一段
- 使用场景 → 典型使用情境
- 产品特色 → 区别于其他产品的亮点
- 常见问题 → FAQ
- 更新日期 → 每次更新需修改
