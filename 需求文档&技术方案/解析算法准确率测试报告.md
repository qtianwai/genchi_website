### 四、根本原因深度分析

#### 4.1 最关键问题：降级算法（`extract_restaurants_from_video`）未同步优化

**这是 v2.2 准确率未提升的核心原因。**

当 `extract_restaurants_priority`（优先级算法）返回空时，系统会降级调用旧算法 `extract_restaurants_from_video`。但旧算法的 Prompt **没有**方案 A/B/D 的任何优化：
- 没有非美食视频过滤（方案 A）
- 没有强化 null 返回（方案 B）
- 没有店铺名/食物名区分规则（方案 D）

**影响示例**：1、2、3、6、10、15、16 均是降级算法造成的错误识别。

#### 4.2 方案 B（强化 null 返回）在优先级算法中也未完全生效

示例 8、11、13 中，优先级算法本应返回 null（多候选店铺、人名误判），但仍然给出了 high 置信度的结果。说明 Prompt 中的规则描述对 AI 的约束力不够强。

#### 4.3 方案 A（非美食过滤）在优先级算法中生效，但降级算法绕过了过滤

示例 15、16 中，优先级算法正确返回了空（非美食视频），但降级算法没有过滤，仍然识别出了内容。

#### 4.4 方案 D（区分店铺名/食物名）在优先级算法中部分生效

示例 7 从 ✗ 变为 ✓，说明方案 D 有效果。但示例 1 仍然失败（降级算法识别了食物名）。

#### 4.5 方案 E（城市降级策略）完全生效

示例 1（南昌）、3（青岛）、4（常州）、6（郫县）的城市识别均正确，说明从标题提取城市的逻辑工作正常。

### 五、优化方向（v2.3）

| 优先级 | 问题 | 解决方案 |
|--------|------|----------|
| P0（最高） | 降级算法未同步优化 | 将方案 A/B/D 的 Prompt 优化同步到 `extract_restaurants_from_video`，或直接**废弃降级算法**，优先级算法返回空时直接返回空 |
| P1（高） | 方案 B 约束力不足 | 在 Prompt 中增加更强的反例示范，或改用 few-shot 方式强化 null 返回 |
| P2（中） | 博主回复评论未获取（示例 1、5） | 已有 `poll_comment_replies_for_confidence` 接口，需要在测试脚本中集成评论回复轮询逻辑 |
| P3（低） | 测试脚本匹配逻辑不够宽松 | 增加同音字/近义词匹配（如"烫"≈"汤"），或改用字符相似度判断 |

## 第六轮测试（v2.6）

测试日期：2026-04-01
测试版本：v2.6（回退冗余修复 + 增强评论高频识别 + 优化评论回复调用策略）

### 一、本轮改动说明

本轮不进行实际测试运行（v2.5 已完成 75% 准确率），而是基于用户反馈对 v2.5 的问题进行代码级修正和 Prompt 增强。

#### 1.1 回退 v2.5 的冗余修复

**问题**：v2.5 在测试脚本中新增了 `"点": "店"` 同义字映射，用于处理"思烤家红谷滩点" vs "思烤家红谷滩店"的匹配。

**根本原因分析**：
- `expected_name` 字段硬编码为"思烤家**红谷滩点**"，这是早期测试数据中的错误
- 正确的店铺名称应为"思烤家**红谷滩店**"（博主回复中明确说明）
- v2.5 识别出的名称本身就是正确的"思烤家红谷滩店"（带"店"字）
- `"点→店"` 同义字修复是**冗余的**——如果评论回复接口正常工作，AI 识别的名称本身就是对的，根本不需要同义字匹配
- 引入冗余的同义字映射反而可能引发新的误匹配问题

**修正内容**：
1. 将测试脚本中 `expected_name` 从"思烤家红谷滩点"更正为"思烤家红谷滩店"
2. 删除 `HOMOPHONES` 中的 `"点": "店"` 映射（只保留 `"烫": "汤"`，因为"烫"vs"汤"是真实的同音字差异）
3. 测试准确率不受影响——v2.5 的成功本身就是靠评论回复正常工作，同义字修复只是兜底逻辑

#### 1.2 增强 Prompt：评论高频店铺识别规则

**目标**：解决示例6（周师饭店）的识别问题——评论中高频出现同一店铺名时，即使没有博主确认也应识别。

**根本原因分析**：
- 示例6的解法是"周师饭店整体评论较多，且网友回复侧面佐证"
- 当前算法完全依赖"博主点赞评论 > 博主回复确认"这条链路
- 周师饭店不在这个链路中，因此无法识别

**新增 Prompt 规则**（插入到 `extract_restaurants_priority` 和 `extract_restaurants_with_replies`）：
- 如果博主点赞评论和博主回复中都没有明确的店铺确认
- 但多个高赞评论（点赞数≥20）中反复提到同一个店铺名（如出现2次以上）
- 且有其他网友回复表示认同（如"就是这家"、"没错"等）
- 可以据此提升置信度或给出答案（置信度最多为 medium）

#### 1.3 优化评论回复接口调用策略

**问题**：示例5（最山城人民广场店）识别出了品牌"最山城不改良重庆火锅"，置信度 high，但因缺少分店信息而判定失败。当前策略下 `confidence == "high"` 不会触发评论回复调用。

**优化方案**：
- 当识别出店铺但名称中**缺少分店相关关键词**（"店"、"广场"、"分店"、"路"、"街"）时，也调用评论回复接口获取分店信息
- 示例5的标题是"上海超级巨无敌好吃的不改良重庆火锅"，评论中提到"人民广场店"，优化后应能识别为"最山城不改良重庆火锅（人民广场店）"

**涉及文件**：`backend/main.py`（快速路径 + 后台任务）、`backend/test_parse_accuracy.py`

#### 1.4 增强分店信息精确提取规则

**新增 Prompt 规则**：在 `extract_restaurants_priority` 和 `extract_restaurants_with_replies` 中增加：
- 如果标题或评论中明确提到分店地址（如"人民广场店"、"红谷滩店"），必须包含在 name 字段中
- 如果只识别出品牌名而缺少分店信息，置信度最多为 medium
