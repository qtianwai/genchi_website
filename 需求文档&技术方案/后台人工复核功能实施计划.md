> 文档版本：v1.2
> 创建日期：2026-04-02
> 最后更新：2026-04-03
> 功能优先级：P0

## 一、背景与目标

### 问题背景

当前 AI 解析准确率约 56%，意味着近一半的视频无法正确识别店铺，或识别结果存在偏差。这些"问题数据"会以两种形式存在于系统中：

1. **未识别**：`video_parse_cache.status = 'completed'` 但 `restaurant_id = null`，即 AI 判断无法提取店铺
2. **识别偏差**：AI 提取了店铺名，但实际上是错误的（当前无法自动检测）

此外，即便 AI 识别了店铺，只要未经人工确认，数据质量仍存在不确定性。

这些问题直接影响地图数据质量，进而影响用户信任。

### 功能目标

在现有 iOS App 内新增一个**仅管理员可见的复核入口**，让平台管理员能够：

- 查看所有未经人工确认的店铺数据（按优先级排序）
- 对每条记录做出判断：确认正确 / 修正店铺 / 标记无效
- 复核时可一键跳转至抖音原视频，辅助信息核实
- 修正店铺时支持模糊搜索，快速定位正确店铺
- 复核通过的店铺在地图上展示「已验证」标识，提升用户信任

## 二、产品视角

### 2.1 用户角色定义

| 角色 | 说明 | 权限范围 |
|------|------|---------|
| 普通用户 | 所有注册用户 | 使用地图、关注博主、收藏店铺 |
| 平台管理员 | 由开发者在数据库中手动指定 | 额外拥有复核入口，可修改店铺数据 |

管理员身份通过数据库 `admin_users` 表维护，不对外暴露注册入口。

### 2.2 核心使用流程

```
管理员打开 App
  → 底部 Tab 出现「复核」入口（普通用户不可见）
  → 进入复核列表页（按优先级排序，显示所有未经人工确认的记录）
  → 看到待复核的视频卡片（含视频标题、博主、AI 解析说明、当前状态、优先级标签）
  → 点击某条记录
  → 进入复核详情页
  → 可点击「在抖音中查看」按钮，跳转至抖音 App 查看原视频内容
  → 选择操作：
      ① 确认正确（AI 已识别店铺，人工确认无误）
      ② 确认无店铺（该视频确实没有探店内容）
      ③ 修正店铺：输入关键词 → 触发模糊搜索 → 从候选列表选择 → 确认入库
      ④ 跳过（暂不处理）
  → 返回列表，该条记录从待复核队列移除
```

### 2.3 复核列表的数据来源与优先级

**复核范围：所有未经人工确认的店铺数据**，包括：

| 优先级 | 类型 | 条件 | 说明 |
|--------|------|------|------|
| P0（最高） | AI 未识别 | `restaurant_id IS NULL` | AI 明确无法提取店铺，需人工判断是否有探店内容 |
| P1 | AI 已识别但未验证 | `restaurant_id IS NOT NULL AND verified = false` | AI 有结果但未经人工确认，存在识别偏差风险 |

列表默认按优先级排序（P0 在前），同优先级内按创建时间倒序。

**类型 C：手动触发复核**（未来扩展）
- 用户勘误提交后，进入复核队列
- 当前版本暂不实现，预留字段

### 2.4 复核详情页的抖音跳转

复核时往往需要查看原视频内容才能判断店铺信息是否正确。详情页提供「在抖音中查看」按钮：

- 点击后通过 URL Scheme 跳转至抖音 App 对应视频页
- 抖音视频 URL 格式：`snssdk1128://aweme/detail?aweme_id=<video_id>`
- 若抖音未安装，降级打开抖音网页版：`https://www.douyin.com/video/<video_id>`
- 跳转后管理员可查看视频内容、评论区、博主主页等信息辅助判断

### 2.5 店铺名模糊搜索与分类确认交互

修正店铺时，提供类似地图搜索的体验，并在选定店铺后自动填充分类供人工确认：

```
管理员在输入框输入关键词（如"老四川"）
  → 输入停顿 300ms 后自动触发搜索（防抖）
  → 调用后端 /api/admin/review/search-restaurant 接口
  → 后端调用高德 POI 搜索，返回候选列表（最多 10 条）
  → 每条候选显示：店铺名、地址、高德分类
  → 管理员从列表中点选正确店铺
  → 点选后进入「确认详情」步骤：
      - 店铺名（只读，来自高德）
      - 地址（只读，来自高德）
      - 美食分类（可编辑下拉/输入框，默认值来自高德分类映射）
  → 管理员确认或调整分类后提交入库
```

**高德分类 → 系统分类映射规则：**

高德 POI 返回的 `type` 字段（如"餐饮服务;火锅店;火锅店"）需映射为系统内的简短分类标签（如"火锅"）。映射在后端维护，前端展示映射后的结果，允许管理员手动覆盖。

若搜索无结果，可调整关键词重新搜索，或切换城市范围。

### 2.6 复核结果的展示

- 复核通过并入库的店铺，在地图卡片上展示一个「✓ 已验证」角标
- 普通用户可见该标识，增强信任感
- 复核标记为「无店铺」的视频，不再出现在复核队列中

### 2.7 管理员入口的隐藏方式

- 不在 App Store 说明中提及
- 普通用户登录后，后端接口返回 `is_admin: false`，Tab 不渲染
- 管理员登录后，后端返回 `is_admin: true`，Tab 自动出现
- 无需单独的管理员 App，降低维护成本

## 三、技术视角

### 3.1 数据库变更

#### 新增表：`admin_users`

```sql
-- 管理员用户表
create table if not exists admin_users (
  id         uuid primary key default uuid_generate_v4(),
  user_id    uuid not null unique,  -- 对应 auth 的 user_id
  note       text,                  -- 备注（如：谁的账号）
  created_at timestamptz default now()
);
```

#### `video_parse_cache` 表新增字段

```sql
-- 复核状态：pending（待复核）/ confirmed（已确认无店铺）/ corrected（已人工修正）/ skipped（已跳过）
ALTER TABLE video_parse_cache ADD COLUMN IF NOT EXISTS review_status text DEFAULT 'pending';
-- 复核操作人（管理员 user_id）
ALTER TABLE video_parse_cache ADD COLUMN IF NOT EXISTS reviewed_by uuid;
-- 复核时间
ALTER TABLE video_parse_cache ADD COLUMN IF NOT EXISTS reviewed_at timestamptz;
```

#### `restaurants` 表新增字段

```sql
-- 是否经过人工验证
ALTER TABLE restaurants ADD COLUMN IF NOT EXISTS verified boolean DEFAULT false;
-- 验证时间
ALTER TABLE restaurants ADD COLUMN IF NOT EXISTS verified_at timestamptz;
```

### 3.2 后端 API 变更

所有复核相关接口需要鉴权：请求头携带 `user_id`，后端校验是否在 `admin_users` 表中。

#### 新增接口

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/admin/check` | 检查当前用户是否为管理员，返回 `{ is_admin: bool }` |
| `GET` | `/api/admin/review/list` | 获取待复核列表，支持分页和优先级排序 |
| `GET` | `/api/admin/review/search-restaurant` | 模糊搜索店铺候选（调用高德 POI），返回候选列表 |
| `POST` | `/api/admin/review/confirm-correct` | 确认 AI 识别结果正确，设置 `verified = true` |
| `POST` | `/api/admin/review/confirm-empty` | 确认某视频无店铺，更新 `review_status = 'confirmed'` |
| `POST` | `/api/admin/review/correct` | 人工修正店铺：从候选列表选定后入库，更新关联关系 |
| `POST` | `/api/admin/review/skip` | 跳过某条记录，更新 `review_status = 'skipped'` |

#### `/api/admin/review/correct` 请求结构

```json
{

  "amap_id": "高德 POI ID",
  "restaurant_name": "老四川（春熙路店）",

  "city": "成都市",
  "latitude": 30.657,
  "longitude": 104.065,
  "category": "火锅"
}
```

`category` 字段由前端传入，值为管理员最终确认的分类（默认来自高德映射，可人工覆盖）。

#### 入库时各字段的处理规则

复核「修正店铺」或「确认正确」时，涉及两张核心表的字段更新，规则如下：

**`restaurants` 表**

| 字段 | 来源 | 说明 |
|------|------|------|
| `name` | 高德 POI | 以高德返回的店铺名为准 |
| `address` | 高德 POI | 高德返回的详细地址 |
| `city` | 高德 POI | 高德返回的城市 |
| `latitude` | 高德 POI | 高德坐标（GCJ-02） |
| `longitude` | 高德 POI | 高德坐标（GCJ-02） |
| `amap_id` | 高德 POI | 唯一标识，用于 upsert 去重 |
| `category` | 管理员确认 | 默认来自高德分类映射，允许人工覆盖 |
| `verified` | 系统写入 | 固定设为 `true` |
| `verified_at` | 系统写入 | 固定设为当前时间 `now()` |
| `created_at` | 系统写入 | 首次入库时写入，upsert 时不覆盖 |

> upsert 以 `amap_id` 为唯一键。若该店铺已存在，仅更新 `verified`、`verified_at`、`category`（若管理员有调整）；`name`、`address`、坐标等以高德最新数据为准。

**`video_parse_cache` 表**

| 字段 | 操作 | 说明 |
|------|------|------|
| `restaurant_id` | 更新 | 指向入库后的 `restaurants.id` |
| `restaurant_name` | 更新 | 同步为最终确认的店铺名 |
| `restaurant_address` | 更新 | 同步为高德地址 |
| `restaurant_city` | 更新 | 同步为高德城市 |
| `restaurant_lat` | 更新 | 同步为高德纬度 |
| `restaurant_lng` | 更新 | 同步为高德经度 |
| `restaurant_amap_id` | 更新 | 同步为高德 POI ID |
| `restaurant_category` | 更新 | 同步为管理员最终确认的分类 |
| `review_status` | 更新 | 修正场景设为 `'corrected'`；确认正确场景设为 `'approved'` |
| `reviewed_by` | 更新 | 写入操作管理员的 `user_id` |
| `reviewed_at` | 更新 | 写入当前时间 `now()` |
| `updated_at` | 更新 | 写入当前时间 `now()` |
| `status` | 不变 | 保持原有 `'completed'`，不修改 |
| `parse_reason` | 不变 | 保留 AI 原始解析说明，不覆盖 |
| `data_source` | 不变 | 保留原始来源标记，不覆盖 |
| `api_cost` / `api_cost_note` | 不变 | 与本次复核无关，不修改 |

> `video_parse_cache` 中的 `restaurant_*` 快照字段是为了避免联表查询而冗余存储的，复核修正后必须与 `restaurants` 表保持一致。

**`author_restaurants` 关联表**

| 操作 | 说明 |
|------|------|
| upsert | 以 `(author_id, restaurant_id, video_id)` 为唯一键，确保关联关系存在 |
| `author_id` | 来自 `video_parse_cache.author_id` |
| `restaurant_id` | 来自入库后的 `restaurants.id` |
| `video_id` | 来自 `video_parse_cache.video_id` |

> 若 AI 原先识别了错误店铺，修正后需检查旧的 `author_restaurants` 记录是否需要清理（当前版本暂不自动清理，留待后续处理）。

#### `review_status` 枚举值说明

| 值 | 含义 | 触发场景 |
|----|------|---------|
| `pending` | 待复核 | 默认值，所有新解析记录 |
| `approved` | 已确认正确 | 管理员确认 AI 识别结果无误 |
| `corrected` | 已人工修正 | 管理员替换了店铺 |
| `confirmed` | 已确认无店铺 | 管理员确认该视频无探店内容 |
| `skipped` | 已跳过 | 管理员暂不处理 |

@Published var searchKeyword = ""

        .debounce(for: .milliseconds(300), scheduler: RunLoop.main)

### 3.4 鉴权方案

当前 iOS 端认证依赖 `user_id`（UUID），所有请求通过 Header 传递：

```
X-User-ID: <user_id>
```

后端复核接口统一做如下校验：

```python
async def require_admin(user_id: str = Header(..., alias="X-User-ID")):
    result = supabase.table("admin_users").select("id").eq("user_id", user_id).execute()
    if not result.data:
        raise HTTPException(status_code=403, detail="无权限")
    return user_id
```

## 五、边界与约束

### 暂不实现（留待后续）

- 复核操作的撤销（当前版本一旦复核不可撤销）
- 多管理员协作（当前版本不区分操作人，只记录 `reviewed_by`）
- 复核统计看板（复核数量、准确率趋势等）
- 用户勘误进入复核队列（依赖 P1 用户勘误功能）

### 数据安全

- 复核接口全部需要管理员鉴权，普通用户调用返回 403
- `admin_users` 表不开放 RLS 公开读，仅后端 Service Role Key 可访问
- 管理员账号通过数据库直接维护，不提供 App 内注册入口

### 与准确率优化的关系

- 本功能是准确率优化的**补充**，不是替代
- 复核数据可作为 few-shot 样本，反哺 AI 提示词优化
- 建议复核时同步记录「正确答案」，为后续算法迭代积累标注数据
