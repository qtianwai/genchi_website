# 冷启动博主录入模块实施计划

## 功能定位

冷启动阶段快速积累美食视频数据的管理员工具。管理员粘贴博主的一条示例视频链接，后台自动获取该博主历史视频列表，经 AI 标题过滤后，美食类视频直接进入人工复核队列（跳过完整解析流程），由管理员手动添加店铺。成本仅为 get-user-video-list 分页调用费（¥0.1/次），相比完整解析节省约 97%。

## 核心解决的问题

- 冷启动阶段需要快速积累大量美食视频数据，但完整解析流程成本过高
- 管理员需要一个高效的批量录入工具，而非逐条提交视频链接

## 功能要点

1. App 新增「录入」Tab，仅管理员可见
2. 管理员粘贴博主示例视频链接 + 自定义 max_count（获取历史视频最大数量）
3. 后台异步执行：获取视频列表 → AI 标题过滤 → 美食视频写入 video_parse_cache
4. 成本只有 get-user-video-list 的分页调用费，记录到 author_background_tasks.api_cost
5. 冷启动记录 `status='cold_start'`、`data_source='cold_start'`，复核优先级 P1
6. 录入模块展示已录入博主列表（头像+名称+任务状态），可跳转博主详情
7. 美食视频在复核模块由人工添加店铺

## 技术实现方案

### 数据库变更

- `video_parse_cache.status` 新增枚举值 `cold_start`（text 类型，无需 ALTER TABLE）
- `video_parse_cache.data_source` 新增枚举值 `cold_start`、`overflow`（v14.1 新增：AI 过滤通过但超出 MAX_PARSE_VIDEOS 的溢出视频，与冷启动共用 cold_start 状态，但来源标识不同）
- `author_background_tasks.task_type` 新增枚举值 `cold_start`
- 复核查询条件扩展：`.in_("status", ["completed", "failed", "cold_start", "pending"])`

### 后端 API（3 个新接口）

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/admin/cold-start/submit` | POST | 提交冷启动任务（视频链接 + max_count） |
| `/api/admin/cold-start/authors` | GET | 获取已录入博主列表（分页） |
| `/api/admin/cold-start/task-status/{task_id}` | GET | 查询任务进度（前端轮询） |

### 后台任务流程 `_cold_start_background()`

```
1. fetch_author_videos(sec_uid, max_count) → 获取视频列表 + API 调用次数
2. filter_food_video_titles(video_list) → AI 标题过滤（零 JustOneAPI 成本）
3. 排除已有记录（get_video_cache_by_id 去重）
4. 批量写入 video_parse_cache（status=cold_start, data_source=cold_start）
5. 更新任务状态和成本
```

跳过的步骤（vs 完整解析）：get-video-detail、get-video-comment、AI 提取店铺、高德搜索

### 成本对比

| 场景 | 50 条视频成本 |
|------|-------------|
| 完整解析 | ¥0.3（列表）+ ¥5.0（详情）+ ¥5.0（评论）≈ ¥10.3 |
| 冷启���录入 | ¥0.3（仅列表）|

### iOS 前端

- 新增 Tab：「录入」（仅管理员可见，tray.and.arrow.down.fill 图标）
- 新增页面：ColdStartView（博主列表）+ ColdStartSubmitSheet（提交弹窗）
- 新增 ViewModel：ColdStartViewModel
- 复核模块适配：ReviewItem 增加 data_source 字段，冷启动记录显示「冷启动」标签

### 与现有模块的集成

- **复核模块**：冷启动记录自然进入复核队列（review_status=pending），优先级 P1
- **parse_link 兼容**：用户提交冷启动已录入的视频时，如果未被复核则走正常解析覆盖
- **博主详情**：复用现有 AuthorDetailView，点击博主行可跳转

### 边界情况处理

1. 博主已存在且有 sec_uid → 跳过 get-video-detail 调用，节省 ¥0.1
2. 重复提交 → 检查是否有 pending/running 的冷启动任务，有则拒绝
3. 视频已存在 → get_video_cache_by_id 去重，跳过已有记录
4. sec_uid 为空 → 返回错误提示
5. max_count 校验 → 前端 10~200，后端也做校验
6. 冷启动记录被人工复核后 → status 更新为 completed，不再需要走解析
