# 解析算法优化方案

基于《解析算法准确率测试报告.md》的分析，当前准确率 25%（4/16），以下是针对各问题的优化方案。

---

## 一、问题优先级排序

| 优先级 | 问题 | 影响用例数 | 预期提升 |
|--------|------|-----------|---------|
| P0（最高） | 非美食视频未过滤 | 2 | 避免脏数据入库 |
| P1（高） | AI 过度识别（应为空却强行给结果） | 3 | 减少错误数据 |
| P2（高） | 博主自己的回复评论未获取 | 3 | 命中关键信息 |
| P3（中） | AI 识别食物名而非店铺名 | 3 | 提升识别精度 |
| P4（低） | 城市编码不可靠（博主注册地 ≠ 拍摄地） | 2 | 改善搜索范围 |

---

## 二、具体优化方案

### 方案 A：非美食视频过滤（解决 P0）

**目标**：在 AI 提取店铺前，先判断视频是否为美食探店内容，非美食视频直接拒绝，不入库。

**实现方式**：在 `extract_restaurants_priority` 的 Prompt 中增加第一步判断：

```
第一步：判断这是否是一个美食探店视频。
- 如果视频内容与美食餐厅探店完全无关（如旅游景点、购物、日常生活等），返回：
  {"is_food_video": false, "reason": "非美食探店视频"}
- 如果是美食探店视频，继续提取店铺信息，返回：
  {"is_food_video": true, "name": "...", "city": "...", ...}
```

**涉及文件**：`backend/ai_extractor.py`

**预期效果**：示例 15、16 正确返回空，不入库。

---

### 方案 B：强化"不确定时返回 null"（解决 P1）

**目标**：当评论中有多个候选店铺、无法确定时，AI 应返回 null 而非强行猜测。

**当前问题**：Prompt 中虽然有 `low` 置信度会被过滤，但 AI 仍然会给出 `high` 置信度的错误答案（示例 8、11）。

**实现方式**：在 Prompt 中增加明确的"返回 null 的条件"：

```
以下情况必须返回 null，不要强行给出答案：
1. 评论中出现 2 个以上候选店铺，且没有博主点赞/回复来确认哪一个
2. 标题中只有食物品类描述（如"肥肠粉"、"火锅"），没有具体店铺名
3. 评论中提到的"店铺名"实际上是人名、地名或食物名
4. 无法从现有信息中确定唯一的店铺
```

**涉及文件**：`backend/ai_extractor.py`

**预期效果**：示例 8、11、13 正确返回空。

---

### 方案 C：获取博主自己发布的评论（解决 P2）✅ 已实现

**目标**：从评论列表中过滤出博主自己发布的评论（而非博主点赞的评论），这些评论通常包含最权威的店铺信息。

**当前问题**：`fetch_video_detail_extra` 只获取了 `is_author_digged=True`（博主点赞的评论），但没有获取博主自己写的评论（通过 `uid` 匹配博主 ID）。

**实现方式（v2.1 已实现）**：

通过评论回复接口（`/api/douyin/get-video-sub-comment/v1`）获取热门评论下的回复，其中包含博主的回复。

核心逻辑：
1. 只对美食相关的热门评论获取回复（通过关键词过滤）
2. 轮询评论回复，找到博主回复（`is_author=True`）即停止，最多 3 次
3. 将评论回复作为 P0 最高优先级信息传递给 AI
4. 置信度 medium 时才调用，控制成本（每日上限 100 次）

**涉及文件**：
- `backend/douyin_parser.py`：新增 `fetch_comment_replies`、`poll_comment_replies_for_confidence`
- `backend/ai_extractor.py`：新增 `extract_restaurants_with_replies`
- `backend/main.py`：集成到快速路径和后台任务

**预期效果**：示例 1（思烤家）、示例 4（新北汤逍遥）、示例 5（最山城人民广场店）可以正确识别。

**成本控制**：
- 每次调用 0.1 元，每日上限 100 次（Railway 可配置 `COMMENT_REPLY_DAILY_LIMIT`）
- 预估月成本：90 元（假设每天 30% 视频需要调用）

---

### 方案 D：区分店铺名与食物名（解决 P3）

**目标**：让 AI 明确区分"店铺名"和"食物名/品类名"，避免将食物名当作店铺名。

**实现方式**：在 Prompt 中增加明确的区分规则：

```
店铺名识别规则：
- 店铺名通常是专有名词，如"甘记肥肠粉"、"最山城"、"陈桥老饭店"
- 以下不是店铺名，不要提取：
  * 食物品类描述：如"老成都肥肠粉"（描述食物风格，不是店名）
  * 地名+食物：如"芳草地面馆"（地名+品类，不是具体店名）
  * 人名：如"庞师不是师"（博主名，不是店名）
- 如果只能找到食物品类描述，没有具体店铺名，返回 null
```

**涉及文件**：`backend/ai_extractor.py`

**预期效果**：示例 1、7、10 减少错误识别。

---

### 方案 E：城市信息降级策略（解决 P4）

**目标**：当城市编码不可靠时（博主注册地 ≠ 拍摄地），优先从标题和话题标签中提取城市信息。

**当前问题**：`city_code` 是视频发布时博主的注册城市，不一定是视频拍摄城市。例如示例 3（青岛视频，但城市编码返回"长沙"）。

**实现方式**：
1. 在 AI Prompt 中明确：城市信息优先从标题和话题标签中提取，`city_code` 仅作参考
2. 在 `fetch_video_detail_extra` 中增加从标题提取城市的逻辑（正则匹配常见城市名）

```python
# 从标题中提取城市（优先级高于 city_code）
CITY_NAMES = ["北京", "上海", "广州", "深圳", "成都", "重庆", "杭州", "南京", 
              "武汉", "西安", "青岛", "南昌", "南宁", "常州", "郫县", ...]
title_city = next((c for c in CITY_NAMES if c in video_title), None)
if title_city:
    city_name = title_city  # 标题城市优先
```

**涉及文件**：`backend/douyin_parser.py`

**预期效果**：示例 3（青岛）、示例 4（常州）城市识别正确，高德搜索范围准确。

## 六、v2.3 优化方案（基于实测结果）

### 根本问题

v2.2 的核心缺陷：**优先级算法返回空时，降级到未优化的旧算法**，导致方案 A/B/D 的效果被旧算法抵消。

### 方案 F：废弃降级算法（解决最关键问题）

**目标**：优先级算法返回空时，直接返回空，不再降级调用旧算法。

**理由**：
- 旧算法 `extract_restaurants_from_video` 没有非美食过滤、没有强化 null 返回、没有店铺名区分规则
- 降级后反而引入了大量错误识别（示例 1、2、3、6、10、15、16 均由降级算法造成）
- 优先级算法已经是更完整的实现，旧算法已无存在价值

**实现方式**：在 `main.py` 的快速路径和后台任务中，移除降级调用逻辑：

```python
# 修改前（有降级）
restaurants = await extract_restaurants_priority(...)
if not restaurants:
    restaurants = await extract_restaurants_from_video(...)  # 删除这段

# 修改后（无降级）
restaurants = await extract_restaurants_priority(...)
```

**涉及文件**：`backend/main.py`

**预期效果**：示例 1、2、3、6、10、15、16 的错误识别消失，准确率提升

---

### 方案 G：同步优化旧算法 Prompt（备选方案）

如果需要保留降级算法作为兜底，则需要将方案 A/B/D 的 Prompt 优化同步到 `extract_restaurants_from_video`。

**涉及文件**：`backend/ai_extractor.py`（`extract_restaurants_from_video` 函数）

**v2.6 本轮不进行测试运行**，而是基于用户反馈对 v2.5 代码进行修正和 Prompt 增强，主要包含以下三项改动：

#### 1. 回退 v2.5 的冗余修复

- **问题**：v2.5 新增了 `"点": "店"` 同义字映射，这是冗余的——评论回复正常工作后，AI 识别的名称本身就是正确的"思烤家红谷滩店"，不需要同义字兜底
- **修正**：将 `expected_name` 更正为"思烤家红谷滩店"，删除 `HOMOPHONES` 中的 `"点": "店"` 映射（只保留 `"烫": "汤"`）

#### 2. 增强评论高频店铺识别规则

- **目标**：解决示例6（周师饭店）——评论中高频出现同一店铺名时，即使没有博主确认也应识别
- **新增 Prompt 规则**：如果多个高赞评论（点赞数≥20）中反复提到同一个店铺名（如出现2次以上），且有网友佐证，可以据此识别（置信度最多为 medium）
- **涉及函数**：`extract_restaurants_priority`、`extract_restaurants_with_replies`

#### 3. 优化评论回复接口调用策略

- **问题**：示例5识别出了品牌"最山城不改良重庆火锅"（high），但因缺少分店信息而判定失败。当前 `confidence == "high"` 不会触发评论回复调用
- **优化**：当识别出店铺但名称中缺少分店相关关键词（"店"、"广场"、"分店"、"路"、"街"）时，也调用评论回复接口获取分店信息
- **涉及文件**：`backend/main.py`（快速路径 + 后台任务）、`backend/test_parse_accuracy.py`
