# 视频解析与数据入库技术方案

本文档详细说明抖音视频解析逻辑、数据库架构设计，以及数据入库的业务流程。当涉及解析或入库逻辑调整时，**必须同步更新本文档**。

---

## 一、系统整体架构

```
用户粘贴抖音链接
       │
       ▼
┌─────────────────────────────────────────────────────────┐
│                    /api/parse-link                       │
│                                                         │
│  ① 视频地址缓存命中？ → 直接返回（毫秒级）                │
│       │（否）                                            │
│  ② 解析当前视频（快速路径：只解析标题，不获取评论）        │
│       │                                                 │
│  ③ 存入 video_parse_cache + restaurants 表              │
│       │                                                 │
│  ④ 启动后台任务（BackgroundTasks）                       │
│       │  → 独立事件循环解析博主历史视频                    │
│       │                                                 │
│  ⑤ 立即返回当前视频的店铺结果给前端                        │
└─────────────────────────────────────────────────────────┘
```

## 二、数据库表结构

### 2.1 authors（博主表）

存储抖音博主的基本信息。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | uuid | 主键 |
| douyin_uid | text | 抖音用户数字 ID（唯一） |
| sec_uid | text | 抖音 sec_uid（用于获取博主视频列表） |
| name | text | 博主昵称 |
| avatar_url | text | 头像 URL |
| created_at | timestamptz | 创建时间 |

### 2.2 restaurants（店铺表）

存储餐厅的详细信息。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | uuid | 主键 |
| name | text | 店铺名称 |
| address | text | 详细地址 |
| city | text | 所在城市 |
| latitude | double | 纬度 |
| longitude | double | 经度 |
| amap_id | text | 高德 POI ID（唯一，避免重复店铺） |
| category | text | 美食分类（火锅、烤肉等） |
| created_at | timestamptz | 创建时间 |

### 2.3 author_restaurants（博主-店铺关联表）

多对多关系，记录哪个博主推荐了哪家店铺。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | uuid | 主键 |
| author_id | uuid | 外键 → authors.id |
| restaurant_id | uuid | 外键 → restaurants.id |
| video_id | text | 来源视频 ID（可追溯） |
| created_at | timestamptz | 创建时间 |

**唯一约束**：`unique(author_id, restaurant_id)` — 同一博主不重复关联同一店铺。

### 2.4 video_parse_cache（视频地址缓存表）

解决同一视频不同分享格式重复解析的问题。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | uuid | 主键 |
| video_url | text | 用户提交的原始抖音分享链接（**唯一索引**，精确匹配） |
| video_id | text | 抖音视频数字 ID |
| author_id | uuid | 外键 → authors.id |
| status | text | 解析状态：`pending` / `parsing` / `completed` / `failed` |
| restaurant_id | uuid | 外键 → restaurants.id |
| restaurant_name | text | 店铺名称快照（缓存解析结果，避免联表查询） |
| restaurant_address | text | 地址快照 |
| restaurant_city | text | 城市快照 |
| restaurant_lat | double | 纬度快照 |
| restaurant_lng | double | 经度快照 |
| restaurant_amap_id | text | 高德 ID 快照 |
| restaurant_category | text | 分类快照 |
| error_message | text | 解析失败时的错误信息 |
| created_at | timestamptz | 创建时间 |
| updated_at | timestamptz | 更新时间 |

**唯一约束**：`unique(video_url)` — 每个用户提交的链接只对应一条记录。
**索引**：`idx_video_cache_videoid` — 按 video_id 查询（用于后台任务跳过已解析视频）。

### 2.5 author_background_tasks（博主后台解析任务表）

管理博主历史探店视频的异步解析任务。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | uuid | 主键 |
| author_id | uuid | 外键 → authors.id |
| task_type | text | 任务类型：`full_scan`（首次全量） / `incremental`（增量更新） |
| total_videos | int | 该任务需要处理的总视频数 |
| processed_videos | int | 已处理完成的视频数 |
| status | text | 任务状态：`pending` / `running` / `completed` / `failed` |
| new_restaurants_found | int | 本次任务发现的新店铺数 |
| error_message | text | 失败时的错误信息 |
| started_at | timestamptz | 任务开始时间 |
| completed_at | timestamptz | 任务完成时间 |
| created_at | timestamptz | 创建时间 |

### 2.6 parse_records（解析记录表）

记录博主是否已完成全量解析（避免重复调用 AI）。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | uuid | 主键 |
| author_id | uuid | 外键 → authors.id（唯一） |
| video_count | int | 解析的视频数量 |
| parsed_at | timestamptz | 解析完成时间 |

### 2.7 user_follows / user_favorites

用户关注博主和收藏店铺的关系表（标准多对多）。

## 三、视频解析流程详解

### 3.1 快速路径（当前视频，用户等待路径）

触发条件：用户每次提交新的抖音链接。

```
parse_link(req):
│
├─ ① 检查 video_parse_cache（用原始链接精确匹配）
│   └─ 命中且 status=completed → 直接返回缓存结果（毫秒级）
│
├─ ② parse_douyin_link(url) → 获取视频信息 + 博主信息
│   └─ JustOneAPI share-url-transfer → get-video-detail
│
├─ ③ upsert_author → 存入/更新博主记录
│
├─ ④ upsert_video_cache(url, status='parsing') → 创建缓存记录
│
├─ ⑤ parse_single_video_fast(当前视频)
│   │（并行获取评论和扩展信息，提升识别率）
│   │
│   ├─ asyncio.gather:
│   │   ├─ fetch_video_detail_extra(vid) → 话题标签、城市、博主点赞评论
│   │   └─ fetch_video_comments(vid, max_count=15) → 普通评论
│   │
│   ├─ extract_restaurants_priority(完整P1-P4信息)
│   │   └─ 降级：extract_restaurants_from_video
│   │
│   ├─ batch_search_restaurants → 高德搜索坐标
│   │
│   └─ _save_video_restaurant:
│       ├─ upsert_restaurant（店铺入库）
│       ├─ link_author_restaurant（博主-店铺关联）
│       └─ update_video_cache_restaurant（更新缓存快照）
│
├─ ⑥ follow_author → 自动关注博主
│
├─ ⑦ 启动后台任务（BackgroundTasks）
│   └─ parse_author_all_videos_background(author_id, sec_uid, current_video_id)
│
└─ ⑧ 立即返回结果给前端
```

### 3.2 后台路径（博主历史视频，不阻塞用户）

触发条件：用户新提交链接后，同时启动。

```
parse_author_all_videos_background(author_id, sec_uid, current_video_id):
│
├─ 在独立事件循环中运行（asyncio.new_event_loop）
│   └─ 避免阻塞 FastAPI 主事件循环
│
├─ create_bg_task → 创建任务记录（status=pending）
├─ update_bg_task_started → 更新为 running
│
├─ fetch_author_videos(sec_uid, max_count=30)
│   └─ 排除 current_video_id（已快速解析过）
│
├─ 遍历每个历史视频：
│   │
│   ├─ get_video_cache_by_id(vid)
│   │   └─ 已 completed → 跳过
│   │
│   ├─ upsert_video_cache(status='parsing')
│   │
│   ├─ fetch_video_detail_extra(vid)
│   │   └─ 获取话题标签、城市、博主点赞评论、热门评论
│   │
│   ├─ fetch_video_comments(vid, max_count=10)
│   │
│   ├─ extract_restaurants_priority(P1~P4 完整信息)
│   │
│   ├─ batch_search_restaurants → 高德搜索
│   │
│   ├─ _save_video_restaurant → 入库
│   │
│   └─ update_bg_task_progress(processed, new_found)
│
└─ complete_bg_task → 更新状态为 completed
```

### 3.3 缓存命中路径（已解析过的链接）

触发条件：用户再次提交完全相同的抖音链接。

```
├─ get_video_cache_by_url(raw_url)
│   └─ status=completed → 直接返回 restaurant_* 快照字段
```

## 四、AI 店铺提取算法

### 4.1 优先级策略（extract_restaurants_priority）

信息优先级从高到低：

| 优先级 | 信息来源 | 说明 |
|--------|----------|------|
| P1（最高） | 视频标题 + 话题标签 + 博主昵称 + 城市 | 标题通常直接含店名，标签提供城市/品类 |
| P2（高） | 博主点赞的评论（is_author_digged=True） | 博主认可 = 视频核心内容 |
| P3（中） | 热门评论（is_hot=True） | 高赞评论往往直接说店名 |
| P4（低） | 普通评论（按点赞数降序） | 兜底信息来源 |

**降级策略**：优先级算法未识别到店铺时，降级调用旧算法（`extract_restaurants_from_video`）。

### 4.2 置信度过滤

- `confidence=low` 的结果直接丢弃
- `confidence=medium/high` 的结果才入库

## 五、高德地图搜索

### 5.1 搜索逻辑

`batch_search_restaurants(restaurants: list[dict]) → list[dict]`

1. 输入：AI 提取的店铺列表 `[{"name": "最山城火锅", "city": "上海", ...}, ...]`
2. 高德关键字搜索（keyword=店铺名称，city=城市编码）
3. 取第一条结果（有坐标的才入库）
4. 返回：`[{"name": "...", "address": "...", "latitude": ..., "longitude": ..., "amap_id": "..."}]`

### 5.2 去重策略

- **入库前去重**（`parse_link` 主流程）：同名+同城市只保留一个
- **高德 ID 唯一**（restaurants 表）：`unique(amap_id)` 避免同一店铺被多个博主重复入库

## 七、关键设计决策

1. **视频级缓存 vs 博主级缓存**
   - 原方案是博主级别缓存（入库博主直接跳过所有视频）
   - 新方案是**视频级别缓存**（每个链接精确命中），同一博主不同视频仍分别解析
   - 权衡：更精细的缓存控制，但首次解析仍需处理所有视频

2. **快速路径获取评论和扩展信息（2026-04-01 优化）**
   - 当前视频快速路径会并行获取评论和扩展信息（话题标签、城市、博主点赞评论）
   - 使用 `asyncio.gather` 并行调用，响应时间约 8-12 秒
   - 相比只解析标题，识别率提升约 30-40%，用户体验和准确性平衡

3. **后台任务隔离**
   - 使用 `asyncio.new_event_loop()` 在独立线程运行
   - FastAPI BackgroundTasks + 独立事件循环，避免任务间相互阻塞
   - Railway 部署环境下，独立线程的任务不受 HTTP 请求超时限制

4. **店铺快照字段**
   - `video_parse_cache` 表直接存储 `restaurant_*` 快照字段
   - 缓存命中时无需联表查询，直接返回快照
   - 权衡：数据冗余，但查询性能更高

## 九、调试模式配置

为控制调试期间的 API 消耗，后台任务支持通过环境变量限制解析数量。

### 配置项（backend/.env）

| 变量名 | 默认值 | 说明 |
|--------|--------|------|
| `DEBUG_MODE` | `false` | 是否启用调试模式。`true` 时限制后台解析数量 |
| `DEBUG_MAX_VIDEOS` | `5` | 调试模式下每次后台任务最多解析的视频数 |

### 使用说明

- **调试期间**：`DEBUG_MODE=true`，每次后台任务最多解析 5 条视频，节省 AI API 费用
- **正式上线前**：将 `DEBUG_MODE` 改为 `false` 或删除该行，恢复解析博主全部历史视频

> 注意：调试模式只限制**后台任务**（历史视频批量解析），不影响**当前视频**的快速解析路径。
