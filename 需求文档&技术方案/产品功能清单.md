# 产品功能清单

> 本文档是产品开发的核心牵引文档，分为「已完成功能」和「待开发功能」两张表格。
> 每次涉及功能新增、调整或删除时，必须同步更新本文档，保持两张表格准确反映当前产品状态。

---

## 已完成功能

| 功能名称 | 定位 | 核心解决的问题 | 关键要点 | 技术实现方案 |
|---------|------|--------------|---------|------------|
| 手机验证码登录 | 用户身份入口 | 用户注册/登录，建立个人数据隔离 | OTP 生成、短信发送、验证码校验、user_id 确定性映射 | 阿里云短信 + Supabase Auth，UUID v5 做 phone→user_id 映射 |
| 抖音链接解析 | 内容获取入口 | 从用户粘贴的文本中提取并解析抖音视频信息 | 支持短链/长链，提取视频详情、博主信息、评论、评论回复 | JustOneAPI 对接，douyin_parser.py 封装 |
| 美食视频过滤 | 内容质量控制 | 过滤非美食/探店内容，避免无关数据入库 | AI 对视频标题批量判断，区分美食探店与其他内容 | qwen-plus AI 过滤，ai_extractor.py |
| 餐厅 AI 识别 | 核心业务能力 | 从视频标题、评论、回复中推断出可信的店铺名 | 优先级策略（博主回复>博主点赞评论>高赞评论），低置信度返回空 | qwen-plus + few-shot prompt，结构化 JSON 输出 |
| 高德 POI 标准化 | 店铺数据标准化 | 将 AI 识别的店铺名映射为标准地址和经纬度 | 多轮降级搜索（精确名+城市→核心名+城市→核心名+全国） | 高德地图搜索 API，amap_service.py |
| 数据入库与缓存 | 数据持久化与复用 | 避免重复解析同一视频，节省 API 成本 | URL/video_id 双重缓存，博主-店铺-视频三方关联入库 | Supabase PostgreSQL，video_parse_cache 表 |
| 地图展示店铺 | 核心产品体验 | 将收集的店铺可视化展示在地图上 | 用户定位、店铺标注、多博主头像聚合展示 | SwiftUI + MapKit，MapView.swift |
| 店铺详情卡片 | 信息浏览 | 快速查看店铺基本信息和关联视频 | 展示店名、地址、分类，可跳转抖音视频 | MapView 内嵌卡片，关联 RestaurantVideo 数据 |
| 关注博主 | 社交化内容组织 | 通过关注博主来聚合感兴趣的店铺推荐 | 关注/取消关注，地图按关注关系过滤店铺 | user_follows 表，AuthorsView.swift |
| 收藏店铺 | 个人偏好管理 | 标记想去/去过的店铺，方便后续查找 | 收藏/取消收藏，收藏列表浏览 | user_favorites 表，FavoritesView.swift |
| 导航跳转 | 到店转化 | 一键导航到目标餐厅 | 适配高德/百度/Apple 地图三种导航 | URL Scheme 跳转，Info.plist 配置 |
| 查看关联视频 | 内容溯源 | 查看某店铺被哪些博主推荐过 | 展示关联视频列表，可跳转抖音 | get_videos_by_restaurant 接口，RestaurantVideo 模型 |
| 后台异步解析 | 解析体验优化 | 快速返回当前视频结果，同时后台补全博主历史店铺 | 快速路径先返回，后台任务扫描历史视频，前端轮询进度 | author_background_tasks 表，scheduler 异步任务 |
| 手动补录店铺 | 解析兜底 | AI 识别失败时允许用户手动添加店铺 | 输入店名+城市，走高德搜索标准化后入库 | /api/manual-add-restaurant，ManualAddRestaurantSheet.swift |
| 博主自动更新检测 | 内容自动化 | 自动检测已关注博主是否发布新美食视频 | 记录连续无新视频天数，超阈值自动关闭检测 | scheduler.py，authors 表 auto_update 相关字段 |
| API 成本控制 | 运营成本优化 | 降低每次解析的第三方 API 调用成本 | 缓存优先、冷却期、AI 预过滤、成本字段记录 | video_parse_cache 缓存策略，api_cost 字段记录 |

## 待开发功能

| 优先级 | 功能名称 | 定位 | 核心解决的问题 | 关键要点 | 技术实现方案 | 优先级说明 |
|--------|---------|------|--------------|---------|------------|-----------|
| P0 | 解析准确率持续优化 | 核心算法提升 | 当前准确率约 56%，目标提升至 80%+ | 强化评论回复利用、置信度分级、多信号融合 | ai_extractor.py prompt 优化，few-shot 反例扩充 | 数据质量是产品核心，准确率直接影响用户信任，无依赖可立即开始 |
| P0 | 后台人工复核机制 | 数据质量保障 | 对未解析或低置信度店铺进行人工确认，提升数据可信度 | 后台复核队列、复核操作界面，店铺展示「已确认」标识 | 新增 restaurant_review_queue 表，后台管理页，restaurants 表增加 verified 字段 | 弥补 AI 准确率不足，与准确率优化互补，开发难度低，收益高 |
| P0 | 提交链接时选择入库范围 | 解析灵活性 | 用户提交链接时可选择仅添加该店铺，或 follow 博主所有推荐 | 解析结果页增加「仅添加本店铺」/「关注博主全部推荐」二选一 | /api/parse-link 增加 scope 参数，前端解析结果页新增选择 UI | 改善核心使用流程，改动小但用户感知强，无前置依赖 |
| P1 | 自动更新解析闭环 | 内容自动化完善 | 检测到新视频后自动完整解析并入库，无需用户手动触发 | 调度器发现新视频后复用 parse_single_video_fast 逻辑完成解析 | scheduler.py 补全 TODO，复用现有解析链路 | 依赖已有博主自动更新检测，补全闭环逻辑，开发量小 |
| P1 | 用户勘误功能 | 数据质量共建 | 用户发现店铺名称或地址有误时可提交勘误，激励用户参与数据修正 | 勘误提交即时感谢卡片、后台审核、审核通过后店铺带勘误者标识并推送通知 | 新增 restaurant_corrections 表，后台审核流程，restaurants 表增加 corrected_by 字段，APNs 通知 | 依赖后台复核机制（P0），激励用户参与数据共建，提升数据长期质量 |
| P1 | 用户自建推荐店铺 | 个人内容创作 | 用户可不依赖博主，自己手动创建和维护推荐店铺 | 用户输入店名/城市，走高德标准化后入库，标记为「我的推荐」 | 扩展 /api/manual-add-restaurant，新增 user_created_restaurants 标记，地图区分展示 | 复用手动补录接口，改动小，是个人地图功能的前置依赖 |
| P2 | 个人专属美食地图 | 个人品牌与社交 | 用户将博主推荐和自己推荐整合为个人专属地图，可分享给伙伴 | 选择收录的博主/店铺，生成个人地图页，伙伴可 follow 并接收新推荐通知 | 新增 personal_maps 表，地图分享页，follow 关系扩展支持 follow 用户地图 | 依赖用户自建推荐（P1）和分享功能，开发量较大，是社交传播的核心载体 |
| P2 | 微信登录 | 用户登录扩展 | 降低注册门槛，提升用户转化率 | 微信 OAuth 授权，获取 openid，绑定用户账号 | 微信开放平台 SDK，WechatAuthManager.swift（已有骨架），后端 /api/auth/wechat-login（已有接口） | 代码骨架已有，等待微信开放平台审核，审核通过后可快速上线 |
| P2 | 博主推荐列表页 | 内容发现入口 | 查看某个博主推荐过的所有店铺 | 按博主维度聚合展示店铺列表，支持筛选城市 | 新增 /api/author/{id}/restaurants 接口，新增博主详情页 | 提升内容发现体验，无强依赖，开发量适中 |
| P2 | 店铺评价/备注 | 个人内容沉淀 | 用户记录自己对店铺的评价和去过的感受 | 用户可对收藏店铺添加文字备注、评分、打卡状态 | 新增 user_restaurant_notes 表，收藏详情页扩展 | 依赖收藏功能（已完成），增强用户粘性，开发量适中 |
| P3 | 城市筛选/切换 | 地图浏览体验 | 快速切换查看不同城市的店铺 | 地图页支持城市下拉选择，自动跳转到对应城市 | MapViewModel 扩展城市过滤逻辑 | 体验优化项，数据量大时价值更高，当前阶段优先级较低 |
| P3 | 店铺搜索 | 内容检索 | 在已收集的店铺中快速找到目标 | 按店名/城市/分类搜索，支持模糊匹配 | 新增搜索接口，iOS 搜索栏 UI | 数据量积累到一定规模后价值才显著，当前阶段优先级较低 |
| P3 | 推送通知 | 用户留存 | 博主发布新美食视频时通知用户 | 自动更新检测到新内容后推送 APNs 通知 | APNs + 后端推送服务，结合自动更新闭环 | 依赖自动更新闭环（P1），需 APNs 证书配置，留存价值高但开发成本也高 |
| P3 | 分享功能 | 社交传播 | 将店铺或博主推荐分享给朋友 | 生成店铺分享卡片或链接，支持微信/系统分享 | iOS ShareSheet，生成分享图片或 Universal Link | 个人地图（P2）的前置依赖，单独做分享价值有限，建议与个人地图一起上线 |
| P4 | 多平台内容接入 | 内容来源扩展 | 支持小红书等其他平台的美食内容解析 | 识别小红书链接格式，适配不同平台的解析接口 | 扩展 douyin_parser.py 或新增 xiaohongshu_parser.py | 开发量大，需适配新平台 API，当前抖音内容已足够，待产品稳定后再扩展 |
